# Real-World Integration Example: SEGB in a ROS4HRI Robot

## Introduction

> **Prerequisites:** This guide assumes you have a working **ROS 2** and **ROS4HRI** environment running inside the PAL tutorials Docker container.  If you have not yet completed Parts 0 and 1 of the official tutorial you should do so first; those parts explain how to start the container, configure your webcam and display, and verify that face detection works.

This tutorial explains how to integrate the **Semantic Ethical Glass Box (SEGB)** into a robot built on top of the **PAL Robotics** simulation framework.  Starting from the "emotion mirror" mission controller created in Part 3 of the ROS4HRI tutorial, we will extend the software to generate semantic audit logs and, optionally, connect to a local large language model (LLM) to enhance interaction.  By the end of this guide, your robot will log what it perceives and how it acts, enabling ethical auditing and transparency.

## Part 3 - Baseline Mission Controller: The Emotion Mirror Game

In Part 3 of the ROS4HRI tutorial you build a mission controller that mirrors the user's facial expression on the robot's face and replies with a fixed sentence.  The controller performs three main tasks:

1. **Subscribe to `/intents`** to receive user intents.  When it receives the special intent `RAW_USER_INPUT`, it uses the text-to-speech (TTS) action to reply with a fixed message.
2. **Set up a `HRIListener`** to access detected faces and create a publisher on `/robot_face/expression` to change the robot's facial expression.
3. **Run a periodic loop** that checks for a new face and, if the expression changes, says "you look ... Same for me!" and publishes the same expression on the robot's face.

Here is the complete baseline script for reference:

```python
import json

from rclpy.node import Node
from rclpy.qos import QoSProfile
from rclpy.action import ActionClient

from hri_actions_msgs.msg import Intent
from hri_msgs.msg import Expression
from tts_msgs.action import TTS
from hri import HRIListener


class MissionController(Node):
    def __init__(self) -> None:
        super().__init__('app_emotion_mirror')
        # Subscribe to intents
        self._intents_sub = self.create_subscription(
            Intent,
            '/intents',
            self.on_intent,
            10)
        # Set up face listener and expression publisher
        self.hri_listener = HRIListener('mimic_emotion_hrilistener')
        self.expression_pub = self.create_publisher(Expression, '/robot_face/expression', QoSProfile(depth=10))
        self.last_expression = ''
        # TTS client
        self.tts = ActionClient(self, TTS, '/say')
        self.tts.wait_for_server()
        # Periodic callback at 10 Hz
        self._timer = self.create_timer(0.1, self.run)

    def on_intent(self, msg: Intent) -> None:
        if msg.intent == Intent.RAW_USER_INPUT:
            # Always say the same thing
            goal = TTS.Goal()
            goal.input = "I'm the famous emotion mirroring robot!"
            self.tts.send_goal_async(goal)

    def run(self) -> None:
        faces = list(self.hri_listener.faces.items())
        if faces:
            face_id, face = faces[0]
            if face.expression and face.expression != self.last_expression:
                expression = face.expression.name.lower()
                # Reply via TTS
                goal = TTS.Goal()
                goal.input = f"you look {expression}. Same for me!"
                self.tts.send_goal_async(goal)
                # Mirror the expression on the robot
                msg = Expression()
                msg.expression = expression
                self.expression_pub.publish(msg)
                self.last_expression = face.expression
```

> **Note:** This simple implementation does not handle any intents beyond raw user input and does not record any events.  It serves as our starting point.

## Part 4 - Optional Integration of a Language Model

The PAL application model recommends separating dialogue into its own node and keeping the mission controller focused on coordination.  However, for demonstration purposes you can integrate a large language model directly.  To do this:

1. **Install an LLM backend.**  The tutorial uses *Ollama*, a local server that exposes a unified API for multiple models.  Follow the installation instructions on the official site and run `ollama serve`.
2. **Install the Python client** within the container: `pip install ollama`.
3. **Add a client to your code.**  Import the client and instantiate it in your controller.  Then implement a `_reply(text, user_name)` method that sends the user's utterance to the LLM with a system prompt and returns the response.  Use a fallback message if the model is unavailable or errors occur:

   ```python
   try:
       from ollama import Client as OllamaClient
   except Exception:
       OllamaClient = None

   class MissionController(Node):
       # ... other initialisation ...
       def __init__(self):
           # ... previous setup ...
           self.ollama = OllamaClient(host="http://127.0.0.1:11434") if OllamaClient else None

       def _reply(self, text: str, user_name: str) -> tuple[str, str]:
           fallback = "I heard you. Could you tell me a bit more so I can help you better?"
           if not self.ollama:
               return fallback, "fallback"
           system = "You are a friendly social robot. Reply with one short, polite sentence suitable for speech output."
           user = f"User ({user_name}) says: {text}"
           try:
               r = self.ollama.chat(model="phi4:14b", messages=[{"role": "system", "content": system}, {"role": "user", "content": user}])
               content = getattr(r.message, "content", "") if hasattr(r, "message") else r.get("message", {}).get("content", "")
               content = " ".join(content.split()).strip()
               return (content[:280] if content else fallback), ("ollama" if content else "fallback-empty")
           except Exception:
               return fallback, "fallback-error"
   ```

4. **Update `on_intent`.**  Extract the text from `msg.data`, call `_reply()` and use the returned string as the TTS output instead of a fixed sentence.  Leave unhandled intents untouched for now.

> **Warning:** Integrating an LLM directly into your mission controller may block the node if the model takes too long to respond.  In production you should separate the chatbot into its own node and communicate via intents.

## Part 5 - SEGB Instrumentation

To audit what the robot hears, sees and says, we need to send semantic logs to the SEGB.  We will use the `semantic_log_generator` library to build a knowledge graph of events and (optionally) publish it to a SEGB server.

### 5.1 Configure the logger and register models

At the top of your script import the SEGB classes:

```python
from semantic_log_generator import (
    ActivityKind,
    EmotionCategory,
    EmotionScore,
    RobotStateSnapshot,
    SEGBPublisher,
    SemanticSEGBLogger,
)
from semantic_log_generator.namespaces import EMOML, ORO
```

Within your controller class declare constants for the SEGB configuration and create the logger:

```python
class MissionController(Node):
    # SEGB configuration (replace values as needed)
    SEGB_BASE_NAMESPACE = "https://gsi.upm.es/segb/robots/demo_robot/v1/"
    SEGB_ROBOT_ID   = "demo_robot"
    SEGB_ROBOT_NAME = "Demo Robot"
    SEGB_LANG       = "en"
    SEGB_LOCATION   = "location_demo_room"
    SEGB_ENABLE_PUBLISH = True
    SEGB_API_URL    = "http://localhost:5000"
    SEGB_API_TOKEN  = "<your_token>"
    SEGB_API_USER   = "demo_robot"

    def __init__(self):
        super().__init__("emotion_mirror")
        # ... other ROS setup ...
        # Create the semantic logger
        self.segb = SemanticSEGBLogger(
            base_namespace=self.SEGB_BASE_NAMESPACE,
            robot_id=self.SEGB_ROBOT_ID,
            robot_name=self.SEGB_ROBOT_NAME,
            default_language=self.SEGB_LANG,
            namespace_prefix="emotion_mirror",
            compact_resource_ids=True,
        )
        # Register each AI model used
        self.intent_model  = self.segb.register_ml_model("intent_nlu_v1", label="Intent NLU", version="1.0")
        self.emotion_model = self.segb.register_ml_model("emotion_big6_v1", label="Emotion Big6 Classifier", version="1.0")
        self.chat_model    = self.segb.register_ml_model(
            "ollama_dialogue_model",
            label=f"Ollama Dialogue ({self.OLLAMA_MODEL})",
            version="1.0",
        )
        # Optionally set up a publisher if publication is enabled
        self.publisher = None
        if self.SEGB_ENABLE_PUBLISH:
            self.publisher = SEGBPublisher(
                base_url=self.SEGB_API_URL,
                token=self.SEGB_API_TOKEN,
                default_user=self.SEGB_API_USER,
                timeout_seconds=15.0,
                verify_tls=True,
                queue_file="/tmp/emotion_mirror/segb_queue.jsonl",
            )
```

### 5.2 Logging user intents

Modify the `on_intent` method to create semantic events whenever a user speaks:

1. **Ignore non-raw intents.**  Only handle `RAW_USER_INPUT` here; other intents remain for later extensions.
2. **Increment an intent counter** and create a unique event identifier.  Capture the current UTC timestamp.
3. **Extract the utterance and identify the human.**  Parse `msg.data` as JSON to get the `text`, `utterance`, `message` or `object` field.  Derive a user identifier (`hint`) from `msg.source` or a default value.  Use a helper `_human(hint, name)` to register the human in the SEGB if it hasn't been seen before.
4. **Create a shared event.**  Call `self.segb.get_shared_event_uri` with `event_kind="human_utterance"`, the timestamp, subject (human URI), text and modality (e.g. `speech`).
5. **Log a listening activity.**  Use `self.segb.log_activity` with `ActivityKind.LISTENING`, linking to the shared event and indicating the intent model used.
6. **Log the input message** via `self.segb.log_message`, linking it to the listening activity and using the correct language tag.
7. **Generate a reply** via `_reply()` (or the fixed message from the baseline).  Send the reply via TTS and note the source (`"ollama"` or fallback).
8. **Log a response activity.**  Create a `ActivityKind.RESPONSE` activity referencing the listening activity and message, recording which model generated the response (the chat model when using the LLM).
9. **Log the output message** with `self.segb.log_message`, linking it to the response activity and to the previous message.
10. **Log the robot state.**  Use `self.segb.log_robot_state` with a `RobotStateSnapshot` describing the mode (`interactive`), mission phase (`demo`), network status and location.
11. **Publish the graph.**  Call a helper `_publish_segb_safe()` that serialises the graph to Turtle and publishes it via `self.publisher` in a background thread.

### 5.3 Logging emotion recognition

Extend the `run()` method so that each time the robot sees a new facial expression it is logged:

1. **Detect a new expression.**  If the detected expression differs from the last one, increment an expression counter and record the current time.
2. **Get the human URI.**  Use `_human(face_id, face_id)` to register the human if needed.
3. **Create a shared event** with `event_kind="human_expression"`, the timestamp, subject (human URI), text (the expression), modality (`vision`) and unique event id.
4. **Log an emotion recognition activity** with `ActivityKind.EMOTION_RECOGNITION`, referencing the shared event and noting the emotion recognition model.
5. **Log an observation** via `self.segb.log_observation`, recording the label and confidence (1.0 for this example) and linking to the shared event.
6. **Annotate the emotion.**  Call `self.segb.log_emotion_annotation` with the recognized emotion category (`EmotionCategory.from_expression(expression)`) and intensity.
7. **Speak and mirror.**  Compose the utterance `"you look {expression}. Same for me!"`, send it via TTS, and publish the same expression on `/robot_face/expression`.
8. **Log a response activity** describing that the robot mirrors the expression and log the output message.
9. **Log the robot state** as in the intents case and publish the graph.

### 5.4 Helper methods

To keep the main methods readable, implement a few helper functions:

* `def _say(self, text: str) -> None:` - sends a TTS goal.
* `def _human(self, hint: str, display_name: str) -> Any:` - normalises the human identifier and registers the human in the SEGB if necessary, storing it in `self._known_humans`.
* `def _state(self, *, event_id: str, activity: Any, note: str, extra: dict | None = None) -> None:` - builds a `RobotStateSnapshot` and logs it.
* `def _publish_segb_safe(self) -> None:` - serialises the graph to Turtle and, if a publisher is available, publishes it in a separate thread to avoid blocking the callbacks.

## Conclusion and Best Practices

By following this tutorial you have extended a simple emotion-mirroring mission controller with semantic logging and optional language-model responses.  The **Semantic Ethical Glass Box** enables you to audit the robot's perceptions and actions, creating a rich knowledge graph of human-robot interactions.  To ensure robustness and maintainability in a real application:

* Keep the dialogue system in its own node or service; send high-level intents to the mission controller instead of raw text.
* Store SEGB API tokens securely outside of your code.  In a production deployment your SEGB server should be accessible via HTTPS and proper authentication.
* Use the PAL application model to separate your mission controller from skills and tasks; the controller should orchestrate rather than perform heavy computation or dialogue.

With these principles in mind, you can build socially intelligent robots that are both interactive and ethically transparent.
