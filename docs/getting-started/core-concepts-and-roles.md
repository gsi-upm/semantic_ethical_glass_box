# Core Concepts and Roles

## Why This Page Exists

SEGB becomes much easier to use once a few words are clear. This page introduces the vocabulary that appears across the
rest of the documentation, using simple examples instead of formal definitions first.

## A Simple Example To Keep In Mind

Imagine this short interaction:

1. a human says, "I am worried about tomorrow's exam",
2. a robot listens,
3. an emotion model marks the situation as anxiety,
4. the robot answers with a calming sentence.

In a traditional system, you might end up with scattered logs in several places. In SEGB, the same interaction can be
stored as connected facts: who spoke, which robot listened, which model was used, what message was produced, and when it
happened.

That is the main idea behind the concepts below.

## Semantic Log

A semantic log is a structured log represented as RDF instead of plain text.

This matters because RDF does not only store isolated lines. It stores relationships between things. For example:

- this message was generated by that activity,
- that activity was performed by this robot,
- this result came from that ML model,
- these two observations refer to the same real-world event.

In SEGB, semantic logs are usually created with the `semantic_log_generator` Python package.

## Turtle

Turtle, often written as TTL, is one of the text formats used to serialize RDF.

You can think of it as the transport format used by SEGB when logs move from the robot side to the backend. A small
Turtle fragment might say, in a machine-readable way, that:

- a robot exists,
- a human exists,
- an activity happened,
- a message was produced.

You do not need to master Turtle to start using SEGB, but it helps to recognize that the backend receives Turtle and the
`/events` endpoint can export Turtle.

## Knowledge Graph

Once Turtle is inserted, the data is stored as triples in a Knowledge Graph.

In this repository, Virtuoso is the graph database. It keeps the stored interaction history so the backend and the UI
can query it later.

When you open the reports page or the graph explorer, you are not looking at separate files. You are looking at views
built from that graph.

## Report

In SEGB, a report is a fixed UI view derived from data stored in the Knowledge Graph.

Examples:

- participant summaries,
- ML usage summaries,
- emotion timelines,
- robot state timelines,
- conversation history.

Reports are not handwritten notes. They are generated by querying the graph with read-only SPARQL queries and then
rendering the result in the UI.

## Shared Context

Shared context is SEGB's way of saying: "these separate observations probably describe the same real-world event."

A simple example:

- robot A hears a sentence,
- robot B hears almost the same sentence a fraction of a second later.

Instead of treating those as completely unrelated events, SEGB can create or reuse a canonical shared context so that
both observations point to the same underlying situation.

This is especially useful in multi-robot and multi-sensor scenarios.

## The Four Main Runtime Pieces

SEGB is easier to reason about if you map each task to one piece of the stack:

### `semantic_log_generator`

This is the Python package used inside a robot or simulation runtime.

Use it when you want to:

- create RDF logs,
- serialize them to Turtle,
- publish them to the backend.

### Backend API

The backend receives logs, validates them, stores them, and exposes read-only endpoints for inspection.

Use it when you want to:

- insert Turtle,
- export the current graph,
- validate queries,
- inspect server logs,
- work with shared-context endpoints.

### Virtuoso

Virtuoso is the graph database that stores the RDF triples.

You do not usually interact with it directly during the first steps, but it is the persistence layer behind the backend.

### Web UI

The web UI is where you inspect the data from an operator or auditor perspective.

Use it when you want to:

- read reports,
- explore the graph,
- test read-only queries,
- review shared-context cases,
- inspect health and server logs.

## Authentication Modes

SEGB has two practical modes:

### Auth Disabled

If `SECRET_KEY` is empty or unset, authentication is disabled.

This is the easiest mode for local learning and demos. Internally, the backend behaves as if the current user had all
roles.

### Auth Enabled

If `SECRET_KEY` is set, the backend expects an HS256 JWT and enforces roles.

This is the mode you should use when you want realistic access control.

## Roles

SEGB uses three main roles:

### `logger`

Use this role for software that publishes logs.

Typical abilities:

- insert Turtle,
- call `/shared-context/resolve`.

### `auditor`

Use this role for read-only inspection.

Typical abilities:

- export events,
- run read-only queries,
- open `/reports` and `/kg-graph`,
- inspect shared-context statistics.

### `admin`

Use this role for operational and maintenance actions.

Typical abilities:

- everything an auditor can do,
- validate Turtle,
- delete graph content,
- review shared-context cases,
- read backend server logs.

## One Full Example, Mapped To Roles

Imagine this small workflow:

1. a robot runtime generates a semantic log and publishes it,
2. an auditor opens the reports page and reviews the conversation,
3. an admin investigates a backend warning and clears the graph before a new demo.

That maps naturally to:

- `logger` for step 1,
- `auditor` for step 2,
- `admin` for step 3.
