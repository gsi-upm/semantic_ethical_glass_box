# Learn Main SEGB Features (Simple)

## Objective

Understand the main SEGB capabilities with a minimal operational flow and almost no code.

## Prerequisites

- Steps 1-3 of the recommended path completed:
  - [Centralized Deployment](../deployment/centralized.md)
  - [Install `semantic_log_generator`](../package/installation.md)
  - [Basic Use: Post Your First Log](../package/usage.md)
- Python environment with simulation dependencies (`semantic_log_generator`, `pydantic`)
- Run commands from repository root (`semantic_ethical_glass_box/`)

If auth is enabled (`SECRET_KEY` set), export a token before running use cases.  
Recommended for this full tutorial: use an `admin` token so all steps run without role errors.

```bash
export SEGB_API_TOKEN="<jwt>"
```

## Steps

Important: UC-02/03/04 and UC-05 (unless you pass `--skip-delete-all`) clear the configured graph before inserting new data.

### 1) Feature: Data ingestion and KG persistence

This step populates SEGB with a complete dataset so all downstream features have data to operate on.

Run the report-ready dataset loader:

```bash
./.segb_env/bin/python -m examples.simulations.run_use_case_02_report_ready_dataset \
  --publish-url http://localhost:5000 \
  --no-print-ttl
```

Auth enabled:

```bash
./.segb_env/bin/python -m examples.simulations.run_use_case_02_report_ready_dataset \
  --publish-url http://localhost:5000 \
  --token "$SEGB_API_TOKEN" \
  --no-print-ttl
```

What this step validates:

- Data generated by the logger is ingested through backend TTL endpoints.
- Triples are persisted in the configured Virtuoso graph.

Detailed backend behavior: [Basic Use: Post Your First Log](../package/usage.md).

### 2) Feature: Visual exploration in UI

This step verifies that the same persisted graph can be explored from the web UI.

Open:

- `http://localhost:8080/reports`
- `http://localhost:8080/kg-graph`

What this step validates:

- `/reports` renders analytics computed from stored interaction data.
- `/kg-graph` renders entities and relations from the same graph.

Detailed UI workflow: [Web Observability](../operations/web-observability.md).

### 3) Feature: Shared-context auto-match (cross-robot)

This step runs two compatible cross-robot observations to show automatic canonical-context reuse.

```bash
./.segb_env/bin/python -m examples.simulations.run_use_case_03_shared_context_auto_match \
  --publish-url http://localhost:5000 \
  --no-print-ttl
```

Auth enabled:

```bash
./.segb_env/bin/python -m examples.simulations.run_use_case_03_shared_context_auto_match \
  --publish-url http://localhost:5000 \
  --token "$SEGB_API_TOKEN" \
  --no-print-ttl
```

What this step validates:

- First observation is usually `created`.
- Second compatible observation is resolved as `matched` to the same shared context URI.

Resolver details: [Shared Context Resolution](../backend/shared-context.md).

### 4) Feature: Ambiguous context + manual review

This step forces an ambiguous match case and applies a review decision.

```bash
./.segb_env/bin/python -m examples.simulations.run_use_case_04_shared_context_ambiguous_review \
  --publish-url http://localhost:5000 \
  --decision accept \
  --no-print-ttl
```

Auth enabled (token must include `admin` role for review endpoints):

```bash
./.segb_env/bin/python -m examples.simulations.run_use_case_04_shared_context_ambiguous_review \
  --publish-url http://localhost:5000 \
  --token "$SEGB_API_TOKEN" \
  --decision accept \
  --no-print-ttl
```

What this step validates:

- Ambiguous candidates are queued for review.
- Manual decision (`accept`/`reject`) updates case state and resulting context mapping.

Resolver review flow: [Shared Context Resolution](../backend/shared-context.md).

### 5) Feature: TTL validation and controlled insertion

This step validates Turtle before insertion and confirms controlled write/export flow.

```bash
./.segb_env/bin/python -m examples.simulations.run_use_case_05_ttl_validate_insert \
  --publish-url http://localhost:5000 \
  --no-print-ttl
```

Auth enabled (token must include `admin` role):

```bash
./.segb_env/bin/python -m examples.simulations.run_use_case_05_ttl_validate_insert \
  --publish-url http://localhost:5000 \
  --token "$SEGB_API_TOKEN" \
  --no-print-ttl
```

What this step validates:

- `/ttl/validate` catches invalid payloads before persistence.
- Valid payload is inserted and can be exported through `/events`.

Detailed backend behavior: [Basic Use: Post Your First Log](../package/usage.md).

## Validation

- `/reports` and `/kg-graph` show data.
- Use-case commands print JSON summaries with expected status fields.
- `GET /events` returns Turtle after each flow.

Auth disabled:

```bash
curl -s http://localhost:5000/events | head -n 20
```

Auth enabled:

```bash
curl -s http://localhost:5000/events \
  -H "Authorization: Bearer <auditor_or_admin_jwt>" | head -n 20
```

## Troubleshooting

- `401/403` in use-case scripts: pass `--token "<jwt>"` with correct role.
- Empty UI views: rerun UC-02 and refresh pages.
- Shared-context flows fail: ensure backend is ready (`/healthz/ready`) before UC-03/UC-04.
